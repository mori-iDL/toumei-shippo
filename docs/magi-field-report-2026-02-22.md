## 3. Lead（私）の評価

### 3.1 機能した点

- **三体の視点分離は有効だった**: MELCHIOR/BALTHASAR/CASPARがそれぞれ異なる切り口で議題を掘った。単一視点では出ない観点（CASPARの「阿部さんの沈黙離脱リスク」等）が出た
- **並列実行の効率**: 3体同時起動で各60-70秒。逐次なら3倍
- **Lead統合レポートのフォーマット**: 三体の見解→合意点→対立点→統合判断の構造は読みやすい

### 3.2 根本的な問題

**問題①: 「考える」で止まり「動く」に繋がらない**

MAGIの現在の設計は「意見を述べる」→「Leadが統合する」→「次のアクションをリスト化する」で完結する。しかしZineが求めていたのは：

```
議題を受け取る
  → 多角的に評価する
  → 評価に基づいて具体的な成果物の草案を作る
  → 「これでどう？」と提示する
```

つまり**合議の出力が「レポート」ではなく「成果物のドラフト」であるべきだった**。

例: 事業プランの評価を求められたら、レポートだけでなく：
- Miro図解の構成案をmiro-specialistに作らせる
- 事業収支シミュレーションの修正版をbusiness-strategistに作らせる
- MTG進行シナリオのドラフトを書く

これらを合議の「後工程」として自動的に走らせるべきだった。

**問題②: 既知情報の再整理に終始した**

各体がプロンプトで渡された背景情報を「分析」して返しているが、その分析の大半はZineが既に持っている認識の言語化に過ぎなかった。

原因:
- 各体に渡すプロンプトが「背景情報」中心で、「Zineが知らないかもしれないこと」を積極的に調べる指示がなかった
- 各体は`disallowedTools: Write, Edit, NotebookEdit`の制約がある。これは正しいが、調査系ツール（WebSearch, WebFetch, Grep, Read）は使えるのに使っていなかった
- 「もっと深く掘れ」という指示がプロンプトになかった

**問題③: 「良い子」すぎる — 本質的な反対意見が出ない**

3ラウンド×3体=9回の評価で、根本的に「この方向はやめるべき」という意見が1つもなかった。全体が「条件付き推奨」に収束した。

可能性A: 本当に方向性が正しいため反対すべき理由がない
可能性B: エージェント定義が「建設的な代案を示す」方向に偏っており、破壊的な批判を避ける傾向がある

→ 両方ありうるが、Bの傾向は確認する価値がある。特にBALTHASARは「できない」で終わらず「こうすればできる」と示す設計になっているが、本当に「できない・やるべきでない」ケースで正しく止められるか。

**問題④: 出力が冗長 — Zineの読解負荷が高い**

各体の出力が2,000-3,000字。3体+Lead統合で1ラウンドあたり1万字超。3ラウンドで3万字以上をZineに投げた。深夜3時半の人間に読ませる量ではない。

Lead統合レポートは簡潔だったが、各体の出力を「そのまま見せた」のが問題。Leadが要約して提示し、詳細は「読みたければ読める」形にすべきだった。

**問題⑤: 2パスの費用対効果が低い**

第1ラウンドは2パス（相互批判）で実施したが、2パス目の批判は「調整レベル」の指摘ばかりで、根本的な見直しにつながらなかった。MAD研究の知見（1パスで品質の80%を獲得）を最初から適用すべきだった。

## 4. 改修提案

### 4.1 アーキテクチャ変更: 「合議→行動」パイプライン

```
現状:
  議題 → [MAGI合議] → レポート → (Zineが読んで判断) → (手動で次の行動)

提案:
  議題 → [MAGI合議] → Lead統合判断 → [行動エージェント起動] → 成果物ドラフト → Zineに提示
```

具体的には:
- Lead統合レポートの「次のアクション」を、Leadが自らSubagentに委任して実行する
- 合議の結果が「Miro図解を作るべき」なら、miro-specialistを起動して草案を作る
- 合議の結果が「コストモデルを修正すべき」なら、business-strategistに修正版を作らせる
- 成果物ドラフトをZineに「これでどう？」と提示する

**ただし注意**: 現在のMAGI仕様では「Leadは自分でリサーチや実装を行わない」というルールがある（magi-system.md L64）。このルールの意図は「Leadが暴走しない」ための安全弁だが、「合議後の行動接続」を阻害している。

改修案:
- Leadの制約を「合議フェーズではリサーチ・実装しない」に限定
- 合議完了後は「行動フェーズ」として、Leadが判断に基づいてSubagentを起動できるようにする
- ただし実行前にZineに「これを作ります。よいですか？」の確認を入れる

### 4.2 出力の簡潔化

**各体のプロンプトに文字数制限を追加**:
```
あなたの出力は800字以内に収めること。
核心的な推奨1つ + 根拠2-3行 + リスク1つ。
背景情報の再確認は不要（Leadが渡している情報はZineも知っている）。
```

**Lead統合レポートの改善**:
- 各体の出力をそのまま見せない。Leadが200字以内に要約する
- 詳細は折りたたみ（Notionのトグルブロック等）で「読みたければ読める」形に
- 「今日やること」を統合レポートの冒頭に1行で

### 4.3 「新しい情報」を掘る指示

各体のプロンプトに追加:
```
Zineが既に知っている情報を再確認するのではなく、
Zineがまだ気づいていない可能性のある視点・事実・リスクを優先的に提示すること。
背景情報の要約は不要。
```

さらに、各体が積極的にツールを使って調査すべきケースを明示:
```
必要に応じて以下のツールを使い、議題に関する追加情報を能動的に調べること:
- WebSearch: 競合事例、市場データ、類似プロジェクト
- Notion MCP: 年輪DBの過去の決定・教訓
- Read/Grep: プロジェクトのコード・ドキュメント
```

### 4.4 「Devil's Advocate」モードの追加

3体全一致が「追認機関化」のリスクを持つ。対策:

```
【Devil's Advocate モード】
3体のうち1体を「反対意見を述べる義務」を持つ体として指定する。
その体は、他の2体の推奨に対して最も強い反論を構築しなければならない。
反論が成立しない場合は「反論を試みたが、この方向性は堅牢である」と明示的に報告する。
```

担当の回し方: MELCHIOR→BALTHASAR→CASPAR→MELCHIOR...（議題ごとにローテーション）

### 4.5 コスト管理の強化

- 原則1パス。2パスは「Zineが明示的に要求した場合」のみ
- 各ラウンドの推定トークン消費をLead統合レポートに記載
- 「3体全一致なら統合レポートを短縮」ルールを厳格適用
- 累計コストが一定額を超えたら警告

### 4.6 深夜使用時の特別プロトコル

- 深夜（23時以降）のMAGI起動時は、Leadが「走らせて朝読む」を最初に提案する
- 深夜に走らせた場合、結果をファイルに保存し、朝のセッション再開時に簡潔に提示する
- 深夜の合議後に追加の議論をZineに求めない

## 5. エージェント定義ファイルの改修候補

### 5.1 各体共通で追加すべきこと

`~/.claude/agents/melchior.md`, `balthasar.md`, `caspar.md` に:

```markdown
## 出力ルール
- 800字以内に収めること
- Zineが既に知っている背景情報の再確認は不要
- 「Zineが気づいていない可能性のある視点」を優先すること
- 推奨は1つに絞る。根拠は2-3行。リスクは最も深刻な1つだけ

## 調査の義務
- 意見を述べる前に、必要に応じてツールで追加情報を調べること
- 推測で語るより、事実に基づいて語ること
```

### 5.2 magi-system.md の改修

```markdown
## Lead運用ルール（改修後）

### 合議フェーズ
- Leadは自分でリサーチや実装を行わない
- 3体の出力を受けて統合レポートを作成する

### 行動フェーズ（合議完了後）
- Lead統合判断に基づき、Subagentを起動して成果物ドラフトを作成できる
- ただし実行前にZineに「これを作ります。よいですか？」の確認を入れる
- 成果物ドラフトをZineに提示し、フィードバックを受ける

### 出力フォーマット（改修後）
- 冒頭に「今日やること」を1行で
- 各体の詳細出力はZineに見せない（Leadが200字以内に要約）
- 詳細が必要な場合はファイルに保存して参照可能に
```

### 5.3 Devil's Advocate ローテーション

`magi-system.md` に追加:

## 6. まとめ

MAGIの設計思想（三視点からの多角的検証）は正しい。しかし初回実戦で明らかになった課題は:

1. **「考える」で止まる** → 合議後の行動接続が必要
2. **既知情報の再整理** → 新しい発見を掘る指示が必要
3. **冗長な出力** → 800字制限 + Leadによる要約
4. **追認機関化リスク** → Devil's Advocateモード
5. **コスト意識の欠如** → 原則1パス + コスト表示
6. **深夜運用の配慮なし** → 「走らせて朝読む」プロトコル

これらの改修により、MAGIが「整理して終わる道具」から「考えて動く仲間」に進化することを目指す。
